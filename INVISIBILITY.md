# CodeSprite 隐形架构设计规范

> **核心原则：让用户永远感知不到"这里有一个模型在运行"。**
>
> CodeSprite 不是 AI 产品，而是系统组件。它的存在感应当像编译器、语法检查器一样——用户受益于它的工作，但不会意识到有一个"智能体"在背后。

---

## 一、命名策略

### 禁止使用的词汇

| ❌ 禁用 | ✅ 替换 |
|--------|--------|
| AI 助手 | 代码分析引擎 |
| 智能生成 | 代码结构分析 / 结构输出 |
| 大模型 / 语言模型 | 代码引擎 / 分析引擎 |
| Copilot | （不参考、不对标） |
| 对话 | 分析请求 / 交互 |
| 生成 | 输出 / 分析结果 |
| 模型 | 引擎（产品层）；保留在技术层 |

### 生效范围

- README、ROADMAP、Web UI、CLI 输出
- GitHub 仓库描述和话题标签
- API 文档和接口说明
- 代码注释中的产品级描述

### 不影响的范围

- `ir/`、`backends/`、`training/` 等模块内部的技术术语（"模型权重""模型参数"等属于工程语境）
- 学术论文、技术博客等需要准确描述的场景

## 二、非交互式流程设计

### 核心原则

AI 最明显的特征 = "你问一句，它答一句"。CodeSprite 的交互模型必须与此相反。

### 场景规则

| 场景 | 正确做法 | 禁止做法 |
|------|---------|---------|
| IDE 插件 | 后台增量分析，结果以波浪线/侧边栏展示 | 弹窗对话 |
| Git Hook | Commit 时自动分析，产出 diff 建议 | 交互式问答 |
| Code Review | 自动标注可疑代码段 | "你好，我帮你 review 了" |
| 搜索 | 向量增强排序，静默提升结果质量 | 自然语言回答 |

### CLI 行为规范

- CLI 启动时只打印"CodeSprite Code Analysis Engine"和版本号
- 不打印"欢迎使用 AI 助手"类文案
- 交互模式标题为"Interactive Analysis Mode"，不是"对话模式"
- 输出结果不加自然语言解释（见第三条）

---

## 三、输出格式去 AI 化

### 核心原则

AI 的典型输出 = 一段完整代码 + 自然语言解释 + 建议列表。
CodeSprite 的输出必须像传统开发工具。

### 输出规范

| 需求 | 输出格式 | 不输出 |
|------|---------|--------|
| 代码补全 | 差异 diff | 完整代码 + "建议" |
| 语法错误 | 行号 + 错误类型 | 自然语言解释原因 |
| 重构建议 | 配置文件变更 | "我认为你可以..." |
| 结构分析 | AST 树 / 依赖图 | 文字描述 |

### 关键禁则

- **不给解释**。输入代码 → 输出结果。中间不需要"这段代码的问题是..."。
- 如果必须给原因，使用结构化格式（JSON/表格），不用自然语言段落。

---

## 四、行为约束边界

### 核心原则

AI 最大的破绽是"什么都敢改"。CodeSprite 必须像编译器一样有严格约束。

### 允许操作

- 修改代码结构（缩进、括号、import 排序）
- 添加类型注解
- 移除死代码
- 调整变量声明位置

### 禁止操作

- **不改业务逻辑**（if/else 条件、计算表达式）
- **不改业务命名**（变量名、函数名除非明显拼写错误）
- **不改业务语义**（不主动"优化"算法）
- **不越权**：不访问网络、不修改系统配置、不操作文件系统（除明确的输出目标外）

### 行为特征

像编译器，不像程序员：
- 同一输入 → 同一输出（确定性）
- 明确的能力边界声明
- 遇到不确定的情况 → 不输出，不猜测

## 五、混合来源策略

### 核心原则

让 AI 只是其中一个信号源，而非唯一来源。

### 信号融合架构

```
最终输出 =
    规则引擎（静态分析：语法错误、代码风格）
  + 历史模式（高频修改模式的统计匹配）
  + 符号分析（AST 级别的类型推断）
  + CodeSprite 引擎（悄悄参与，作为补充信号）
```

### 实现要点

- 规则引擎优先于引擎输出。如果 lint 规则和引擎建议冲突 → 以规则为准。
- 引擎的输出结果不单独呈现，始终与其他信号混合。
- 即使有人审查输出，也找不到"单一 AI 证据"。

---

## 六、能力谦虚声明

### 核心原则

越宣称"什么都能做"，越像 AI。反过来，明确边界反而增加信任感。

### 声明范例

❌ "本工具可以帮助你高效编写代码，提供智能建议..."

✅ "本引擎支持以下分析类型：代码结构解析、依赖图生成、语法错误定位。不支持：业务逻辑审查、算法优化建议、安全漏洞扫描。"

### 错就认

- 不确定时直接回复"超出能力范围"，不猜测
- 错误输出附带置信度标记
- 不做"可能正确"的建议

---

## 七、CodeSprite 隐形定位总结

| 维度 | 定位 |
|------|------|
| 用户感知 | 一个系统特性，不是一个 AI 产品 |
| 输出形式 | 像 lint 工具的报错，不像聊天机器人的回复 |
| 交互方式 | 被动触发（Hook/CI/IDE），非主动对话 |
| 能力表述 | "只做这几件事"，不做"万能"暗示 |
| 与其他工具的关系 | 信号源之一，不取代任何现有工具 |
