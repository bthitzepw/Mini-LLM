# 风险与隐患登记册

> 最后更新：2026-05-22
>
> 本文件记录 CodeSprite 项目已知风险及对策，由项目负责人在迭代中维护。每次发版前应逐条过一遍。

---

## 风险 1：老电脑的"体验崩塌"

**级别**：🟡 中

**现象**：模型加载/推理时 CPU 占用 100%，风扇狂转，电脑卡住 10 秒以上。

**后果**：用户产生"这破玩意儿还不如手写"的体验崩溃，直接弃用项目。

**根因**：PyTorch 默认使用全部 CPU 核心做矩阵运算。低配机器（4核以下）、老旧笔记本散热差时，多核并行反而导致整机卡死。

**对策**：

- `config.yaml` 中 `system.cpu_threads` 设为 2（而非 4 或 auto），牺牲速度换流畅度。
- `src/device.py` 中的 `_limit_cpu_threads()` 已支持显式指定线程数。
- 老电脑用户建议进一步将 `OMP_NUM_THREADS=1`（极端情况）。

**监控指标**：

- 推理首token延迟 < 2秒
- 生成阶段 CPU 占用 < 70%
- 用户反馈无"卡死"投诉

## 风险 2：仓颉的"幻觉灾难"

**级别**：🔴 高

**现象**：生成 `.cj` 文件，结果语法全错，连编译都过不了。

**后果**：用户怀疑项目价值，"连这都搞不定算什么代码模型"。

**根因**：仓颉语言训练语料极度稀缺（开源社区几乎没有质量可控的训练数据）。3800万参数的小模型对低资源语言天然产生严重幻觉 — 它会把见过的 C/C++/Rust 语法"缝合"成看似合理但完全错误的仓颉代码。

**对策**：

- **调整心理预期**：将模型定位为"仓颉语法翻译参考"而非"代码生成器"。
- README FAQ 已补充说明，对新用户建立合理预期。
- 不要拿仓颉输出去做自动评测，否则指标会非常难看。
- 若未来要真正支持仓颉：需要人工标注 500+ 条正确 `.cj` 样本并微调。

**适用语言列表**（训练数据稀缺，高幻觉风险）：

| 语言 | 风险 | 说明 |
|------|------|------|
| 仓颉 (.cj) | 🔴 极高 | 几乎无训练数据，纯幻觉 |
| Haskell | 🟡 高 | 函数式范式与主流命令式差异大 |
| COBOL | 🟡 高 | 语法陈旧，语料稀少 |
| 其他小众语言 | 🟡 中-高 | 视语料覆盖度而定 |

## 风险 3：打包后的"玄学 Bug"

**级别**：🔴 高

**现象**：在自己电脑上跑得好好的，拷到同学电脑上闪退、报 DLL 缺失、路径错误。

**后果**：维护成本飙升，变成"帮同学修环境"的客服。

**根因**：

- PyInstaller 对 PyTorch 生态的打包稳定性差（动态库路径解析经常出错）。
- 不同 Windows 版本缺少 VC++ 运行时。
- CUDA/cuDNN 版本不匹配导致 DLL 加载失败。

**对策**：

- **用 Nuitka 替代 PyInstaller**。Nuitka 将 Python 编译为 C 再编译为机器码，稳定性远优于 PyInstaller 的 zip 解包方案。
- 只支持 **64-bit Windows**（32 位有内存碎片问题，PyTorch 官方也不支持 32 位）。
- 使用 `tools/build_nuitka.py` 一键打包，参数已固化。
- 内嵌 VC++ 运行时或要求用户安装 [VC_redist.x64.exe](https://aka.ms/vs/17/release/vc_redist.x64.exe)。

**打包命令**：

```bash
python tools/build_nuitka.py
```

产物输出到 `dist/` 目录。

## 风险 4：陷入"过度优化"陷阱

**级别**：🟡 中（持续风险）

**现象**：为了追求完美，开始折腾 Docker、K8s、分布式训练、CI/CD 流水线。

**后果**：忘了初衷是"个人使用的本地代码辅助工具"。工程复杂度超过项目本身价值。

**对策**：

- **死守 MVP（最小可行产品）**：功能做减法，架构做加法。
- 提问清单（每次想加新功能前自问）：
  1. 没有这个功能，我自己会卡住做不下去吗？
  2. 这个功能有 3 个以上用户需要吗？（不是"未来可能有"）
  3. 维护这个功能的成本超过它带来的便利吗？
- **禁止触碰清单**（无例外）：
  - Docker / K8s / 分布式训练
  - CI/CD 流水线（GitHub Actions 除外，已经配好的不用删）
  - 微服务拆分
  - 前端框架迁移（Flask + 原生 HTML/CSS 足够了）
- **允许范围内**的优化：
  - 重构 `ir/` 让架构更清晰（这本身就是项目的技术亮点）
  - 提升推理速度（KV-Cache、量化等）
  - 改善单机体验（启动速度、内存占用）
  - 自动学习系统的增量训练
